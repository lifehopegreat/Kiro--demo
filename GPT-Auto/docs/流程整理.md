## 一、整体目标

脚本围绕一个浏览器自动化流程运行：启动指定 Roxy 浏览器环境，完成账号创建，获取登录态，打开订阅付款链路，准备支付资料，并在失败时记录截图和状态，方便下一轮排查。

## 二、主流程

入口函数：`main.py` 中的 `main()`。

单账号处理函数：`process_one_account()`。

执行顺序：

1. 启动 Roxy Profile
   - 从配置读取 `roxy_profile_id`。
   - 通过 `roxy_client.launch_roxy_profile()` 启动浏览器环境。
   - Playwright 通过 CDP 连接浏览器。

2. 初始化浏览器上下文
   - 新建浏览器上下文。
   - 清理 cookies。
   - 打开新页面。

3. 生成并占用邮箱
   - 邮箱前缀按序号生成，例如词库前缀加三位序号。
   - 会检查这些文件中是否已使用过邮箱：
     - `used_emails.txt`
     - `accounts.txt`
     - `payment_links.txt`
     - `automation.log`
     - `run_state.json`
   - 新邮箱写入 `used_emails.txt`，并更新 `run_state.json`。

4. 打开 ChatGPT 注册页
   - 访问 `chatgpt.com`。
   - 点击注册入口。
   - 输入邮箱并继续。

5. 获取邮箱验证码
   - 点击继续前记录邮箱 UID 基线。
   - 使用 IMAP 轮询最新验证码邮件。
   - 优先匹配目标邮箱相关邮件。
   - 获取验证码后写入运行状态。

6. 完成基础资料
   - 输入验证码。
   - 填写随机英文姓名。
   - 填写年龄。
   - 提交创建账号。

7. 获取登录 token
   - 等待注册完成。
   - 从页面环境里尝试获取访问 token。
   - 写入 `accounts.txt` 的账号状态为 `Free`。
   - `run_state.json` 中只应保留脱敏 token。

8. 生成并打开付款链接
   - 使用 token 进入付款链接生成页面。
   - 尝试生成付款链接，失败会重试。
   - 打开生成的付款页面。
   - 校验 `Total due today` 是否为 0.00。

9. 准备卡信息
   - 优先可以走卡密兑换。
   - 如果存在 `manual_card_info.json`，会跳过卡密兑换，直接使用手动卡信息。
   - 卡密会用 `used_cardmi.txt` 做去重和占用记录。
   - 当前日志里多次失败点集中在卡密兑换页显示失败。

10. 填写 PayPal 订阅页
    - 选择 PayPal 付款方式。
    - 勾选订阅协议。
    - 点击订阅并等待进入 PayPal 后续页面。

11. PayPal 创建账号和支付资料
    - 进入创建 PayPal 账号页面。
    - 填写邮箱、手机号、卡信息、账单地址、姓名、密码等。
    - 如果出现安全挑战，会截图保存并中断当前流程。

12. 成功收尾
    - 标记卡密使用完成。
    - 将账号状态追加为 `Plus`。
    - 关闭浏览器，进入下一轮。

## 三、循环与异常处理

`main()` 会循环执行 `process_one_account()`：

- 如果返回成功：计数加一，更新 `run_state.json`，等待数秒后处理下一个账号。
- 如果返回失败：退出循环。
- 如果抛出 `RestartAccountRequired`：记录原因，等待后重新开始一个账号。
- 普通异常：写入 `run_state.json`，保存错误截图，返回失败。

常见异常产物：

- `error_*.png`：普通异常截图。
- `restart_*.png`：需要重开账号时的截图。
- `paypal_security_challenge_*.png`：PayPal 安全挑战截图。
- `card_debug.txt`：卡密兑换页失败时保存的调试文本。

## 四、关键文件

配置与代码：

- `main.py`：主自动化流程。
- `roxy_client.py`：Roxy API 封装。
- `config.json`：运行配置，含敏感密钥和邮箱应用密码。

输入数据：

- `cardmi.txt`：卡密池。
- `manual_card_info.json`：手动卡资料。
- `phone.txt`：手机号和短信链接信息。

状态与结果：

- `run_state.json`：当前流程状态，可能含支付链接、手机号、密码等敏感运行数据。
- `state.json`：辅助状态。
- `used_emails.txt`：已占用邮箱。
- `used_cardmi.txt`：已使用卡密。
- `accounts.txt`：账号结果记录，历史内容中包含高敏 token。
- `cards_result.jsonl`：卡信息结果记录。
- `payment_links.txt`：支付链接记录。
- `automation.log`：完整运行日志，可能包含高敏运行信息。

## 六、排查顺序

建议按这个顺序排查：

1. 先看 `run_state.json`
   - 确认最新 `step`、`email`、`error`。
   - 注意不要把里面的敏感字段发给外部。

2. 再看 `automation.log` 末尾
   - 定位最近一次启动时间。
   - 找到最后一条 `ERROR` 或 `WARNING`。

3. 如果是卡密问题
   - 检查 `cardmi.txt` 是否还有可用卡密。
   - 检查 `used_cardmi.txt` 是否已把当前卡密标记为占用。
   - 查看 `card_debug.txt` 中兑换页面的错误提示。

4. 如果是 PayPal 安全挑战
   - 查看对应的 `paypal_security_challenge_*.png`。
   - 这类问题通常需要人工处理或更换环境资料。

5. 如果是验证码问题
   - 检查 Gmail IMAP 是否可用。
   - 检查域名邮箱转发是否正常。
   - 检查 UID 基线之后是否有新验证码邮件。

6. 如果是付款链接问题
   - 多数是网络或页面响应问题。
   - 日志中已有重试机制，先看是否最终生成成功。

## 七、敏感信息处理建议

当前项目里有多处高敏信息，建议后续整理时优先做脱敏和隔离：

- 不把真实 token 写入 `accounts.txt`。
- 不把 PayPal 密码、手机号、完整卡号、完整地址长期保留在 `run_state.json`。
- 将 `config.json` 中的密钥迁移到本机环境变量或单独的未跟踪密钥文件。
- 对历史日志和结果文件做一次脱敏归档。
- 调试截图保留必要时间后清理，避免包含页面个人信息。
